# 发型设计网站开发过程分析与补充文档

本文档用于重新梳理当前发型设计网站的开发过程、已有决策、当前代码状态、接口接入方式、已发现问题和后续需要补充确认的事项。

## 1. 项目目标

做一个细分方向的 AI 图片工具网站，方向是“发型设计”。

用户上传半身照片，填写或选择发型需求，网站把用户输入和内置提示词组合后，调用 MoleAPI 中转站生成图片。生成结果在页面展示，用户可以下载，也可以主动公开到作品墙。

第一版重点不是复杂账号系统，而是先跑通：

- 中英双语页面
- 发型设计工具页
- API Key 设置
- 上传半身照片
- 调用 MoleAPI 生成图片
- 展示生成结果
- 下载图片
- 公开作品墙

## 7. MoleAPI 接入方式

文档地址：

https://docs.moleapi.com/zh-CN/docs/getting-started

当前使用 MoleAPI 的 OpenAI 兼容方式。

Base URL：

```text
https://api.moleapi.com/v1
```

当前请求接口：

```text
POST /images/edits
```

完整地址：

```text
https://api.moleapi.com/v1/images/edits
```

请求头：

```text
Authorization: Bearer <用户填写的 API Key>
```

请求体：

```text
multipart/form-data
```

表单字段：

- `model`
- `prompt`
- `image`
- `size`
- `n`

当前默认：

```text
model: gpt-image-2
size: 1024x1024
n: 1
```

## 8. 当前提示词逻辑

用户可以输入自由文本，也可以选择快捷标签。

快捷标签包括：

- 短发
- 长发
- 卷发
- 刘海
- 自然棕
- 商务
- 温柔
- 韩系
- 日系
- 自然真实

系统会组合内置提示词。

当前提示词核心要求：

- 基于上传的半身照片生成发型预览
- 保留同一个人的脸、身份、表情、肤色、身体姿态
- 只改变发型和发色
- 发型要自然、真实、可落地
- 适合作为真实理发参考
- 不改变五官
- 不改变年龄
- 不改变性别表达
- 不大幅改变体型或背景

当前英文提示词结构大致为：

```text
Create a realistic hairstyle preview based on the uploaded half-body portrait.
Keep the person's face, identity, expression, skin tone, and body pose consistent.
Only redesign the hairstyle and hair color according to the request.
Make the hairstyle natural, wearable, and suitable for a real salon reference.
Avoid changing facial features, age, gender, body shape, or background drastically.
Selected style directions: ...
User request: ...
```

后续可以进一步优化：

- 针对不同发型类型写更细的提示词
- 根据性别、年龄、脸型自动补充建议

- 增加负面提示词

## 9. MoleAPI 返回处理

实际返回示例中，MoleAPI 返回结构为：

```json
{
  "data": [
    {
      "url": "",
      "b64_json": "很长的 base64 字符串",
      "revised_prompt": "..."
    }
  ],
  "created": 1778475828,
  "background": "auto",
  "output_format": "png",
  "quality": "auto",
  "size": "1024x1024"
}
```

最开始页面只识别图片 URL，所以报错：

```text
生成失败，请检查 MoleAPI Key、Base URL、模型名称或第三方服务状态。
```

后来修正为支持：

- `data[0].b64_json`
- `b64_json`
- `base64`
- `image_base64`
- `data:image/png;base64,...`
- `url`
- `imageUrl`
- `image_url`
- 嵌套对象中的图片字段
- 数组中的图片字段

当前代码处理方式：

1. 如果返回是图片二进制，则转成浏览器可显示的图片地址
2. 如果返回是 JSON，则先完整读取文本
3. 再解析 JSON
4. 递归查找图片字段
5. 如果找到 base64，则转成：

```text
data:image/png;base64,...
```

## 10. 已遇到的问题

### 10.1 终端中文显示乱码

PowerShell 输出中文时出现乱码。

判断：

这主要是终端编码显示问题，不一定代表文件内容损坏。

已做检查：

- 用 UTF-8 读取文件确认关键中文存在
- 页面实际应按 UTF-8 解析

### 10.2 本地环境没有可用 Python

尝试启动：

```powershell
python -m http.server 4173
```

失败，原因是系统找不到 `python`。

当前解决：

项目是静态网站，可以直接双击 `index.html` 测试。

### 10.3 Node 被系统拒绝运行

尝试：

```powershell
node --check app.js
```

失败，原因是系统拒绝访问。

影响：

没有做 Node 语法检查。

替代：

- 通过静态检查关键字符串
- 尽量保持代码简单
- 后续建议在正常开发环境中用浏览器控制台检查错误

### 10.4 MoleAPI 返回 base64，但页面没有识别

问题：

接口返回 `data[0].b64_json`，页面没有提取。

已修复：

支持读取 `b64_json` 并转成图片。

### 10.5 请求时间过长

用户反馈：

```text
CAUTION: request is not finished yet!
13.1 min
```

最初判断：

可能是接口长时间没有结束。

后来重新判断：

也可能是接口已经返回，但前端没有完整接收或正确解析。

已调整：

- 不再直接 `response.json()`
- 先 `response.text()` 完整读取响应
- 再解析图片
- 超时时间从 3 分钟改为 15 分钟

当前仍需观察：

- MoleAPI 是否真实需要这么久
- 浏览器直连是否稳定
- 返回体过大时浏览器是否卡顿
- 是否需要改成服务端中转

## 11. 当前架构风险

### 11.1 浏览器直连 MoleAPI 的风险

当前是浏览器直接请求 MoleAPI。

风险：

- 可能遇到跨域限制
- API Key 暴露在浏览器请求里
- 无法统一记录生成任务
- 无法可靠保存结果图
- 无法做作品墙服务端化
- 无法做重试和任务恢复
- 长时间请求容易被浏览器中断

建议正式版本改成：

```text
浏览器 -> 自己的后端 -> MoleAPI -> 自己的后端 -> 浏览器
```

### 11.2 没有服务端存储

当前作品墙只存在本地浏览器。

风险：

- 换设备不可见
- 清除缓存后丢失
- 无法形成真实公开作品墙
- 无法后台下架违规作品

正式版本需要：

- 数据库
- 图片存储
- 作品公开状态
- 管理后台

### 11.3 API Key 本地保存风险

当前 API Key 存在浏览器本地。

优点：

- 不需要注册
- 不需要服务器保存 Key
- 第一版简单

风险：

- 用户本机被他人使用时可能泄露
- 浏览器插件可能读取
- 如果后续有后端，需要避免记录完整 Key

建议：

- 页面只显示部分 Key
- 提供一键清除
- 后端日志不记录 Key
- 后续考虑加密保存或只会话保存
